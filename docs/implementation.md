# Booking Handling Implementation Plan

## 1. Objective

Implement server-side booking handling for NUS4STAY that:

- Prevents double booking under concurrent requests.
- Holds room inventory while payment is pending.
- Expires unpaid bookings after 30 minutes.
- Restores `room_availability` exactly once after expiry or cancellation.
- Marks the booking code as expired without deleting or reusing it.
- Follows [`rpc-function.md`](rpc-function.md) and [`security.md`](security.md).
- Removes booking authority from the React client and `localStorage`.

The database is the source of truth. A frontend countdown is only a display of `expires_at`; it must never decide whether a booking is still valid.

## 2. Current State and Gaps

The current frontend behavior is not safe for production booking:

- `src/services/db.js` creates bookings in `localStorage`.
- The booking code is generated in the browser.
- Booking status can be changed from the browser.
- `PendingPayment.jsx` uses a local 24-hour timer instead of a database timestamp.
- “Cek Status Pembayaran” currently changes a booking directly to confirmed.
- There is no transactional inventory hold.
- Two simultaneous clients can reserve the same remaining room.

These paths must be replaced, not kept as a fallback after the RPC implementation is enabled.

## 3. Required Business Decisions

The implementation will use the following rules.

### 3.1 Date semantics

Booking dates use a half-open range:

```text
[check_in, check_out)
```

The check-in date consumes inventory. The check-out date does not. Therefore, one guest may check out on the same date another guest checks in.

### 3.2 Inventory model

`rooms` represents a bookable room type. `room_availability` stores the number of available units for each room type and stay date.

If every `rooms` row represents one physical room, set `total_units = 1`; the same mechanism still works.

Global room state such as `rooms.is_active` must not be toggled during booking or expiry. Availability is date-specific.

### 3.3 Booking hold

- A successful `create_booking()` call immediately reserves one unit for every night.
- Initial booking status: `pending_payment`.
- Initial payment status: `unpaid`.
- `expires_at` is generated by PostgreSQL as `now() + interval '30 minutes'`.
- The client must not send or override `expires_at`.

### 3.4 Expired booking code

- The booking record remains in the database.
- Its status becomes `expired` and `expired_at` is recorded.
- The booking code remains unique and is never reused.
- Payment, approval, and booking-detail RPCs must reject or identify the expired code.

### 3.5 Manual payment proof

Recommended rule for the current manual-transfer flow:

- Proof uploaded before `expires_at` changes the booking to `payment_review` and payment to `submitted`.
- A booking in `payment_review` is not expired by the 30-minute unpaid-booking job because the guest submitted payment on time.
- Approval changes it to `confirmed` / `paid`.
- Rejection changes it back to `pending_payment` / `rejected` and gives a new 30-minute deadline, or releases it immediately if that is the chosen business policy.

This is a required refinement to the current `rpc-function.md`. Saving only a proof URL while leaving the booking in `pending_payment` could expire a legitimately paid booking while an admin is still reviewing it.

For a payment gateway, only a verified server-to-server webhook may set `payment_status = paid`.

## 4. Booking State Machine

```text
                                  verified payment
pending_payment ------------------------------------------------> confirmed
      |                                                              |
      | proof uploaded before deadline                               | stay lifecycle
      v                                                              v
payment_review ---------------- approve ------------------------> confirmed
      |
      | reject
      v
pending_payment (new deadline) or cancelled

pending_payment -- expires_at reached and still unpaid --> expired
pending_payment -- guest cancellation ------------------> cancelled
```

Inventory remains reserved for `pending_payment`, `payment_review`, and `confirmed` bookings. Inventory is released once for `expired` or `cancelled` bookings.

Suggested migration structure:

### 5.1 `rooms`

Add or confirm:

| Column | Type | Rule |
| --- | --- | --- |
| `inventory_count` | integer | `NOT NULL`, greater than 0, default 1 |
| `is_active` | boolean | Must be true to create a booking |

### 5.2 `room_availability`

One row represents inventory for one room type on one date.

| Column | Type | Rule |
| --- | --- | --- |
| `room_id` | uuid | FK to `rooms.id` |
| `stay_date` | date | A consumed night |
| `total_units` | integer | Must be greater than 0 |
| `available_units` | integer | Between 0 and `total_units` |
| `updated_at` | timestamptz | Database maintained |

Constraints:

```text
PRIMARY KEY (room_id, stay_date)
CHECK (total_units > 0)
CHECK (available_units >= 0)
CHECK (available_units <= total_units)
```

Availability rows should be generated ahead of time, for example for the next 365–540 days. A missing row must cause booking creation to fail with `AVAILABILITY_NOT_INITIALIZED`; it must not be interpreted as unlimited inventory.

### 5.3 `bookings`

Add or confirm:

| Column | Type | Notes |
| --- | --- | --- |
| `id` | uuid | Primary key, database generated |
| `booking_code` | text | Unique, server generated, never reused |
| `user_id` | uuid | Always derived from `auth.uid()` |
| `room_id` | uuid | FK to `rooms.id` |
| `check_in` | date | Inclusive |
| `check_out` | date | Exclusive |
| `guest_count` | integer | Positive and within room capacity |
| `unit_price` | numeric | Price snapshot from the room record |
| `night_count` | integer | Derived server-side |
| `total_price` | numeric | Derived server-side |
| `booking_status` | text/enum | State machine controlled |
| `payment_status` | text/enum | State machine controlled |
| `expires_at` | timestamptz | Set by database for pending payment |
| `expired_at` | timestamptz | Set when expired |
| `inventory_released_at` | timestamptz | Idempotency guard for inventory release |
| `idempotency_key` | uuid | Prevents duplicate booking from repeat submission |
| `created_at` | timestamptz | Database generated |
| `updated_at` | timestamptz | Database maintained |

Required constraints and indexes:

- `UNIQUE (booking_code)`.
- `UNIQUE (user_id, idempotency_key)`.
- `CHECK (check_out > check_in)`.
- Partial expiry index on `(expires_at)` where `booking_status = 'pending_payment'`.
- Index on `(room_id, check_in, check_out)`.
- Index on `(user_id, created_at DESC)`.

Booking dates, room ID, and price snapshots must be immutable after creation. Changing dates requires cancelling/releasing the old booking and creating a new booking.

### 5.4 `payments`

Add or confirm:

- `booking_id` FK.
- `proof_url` for manual transfer.
- `provider_event_id` with a unique constraint for gateway webhook idempotency.
- `status`: `unpaid`, `submitted`, `paid`, `rejected`, or `late_payment`.
- `submitted_at`, `paid_at`, `reviewed_at`, and `reviewed_by`.

## 6. `create_booking()` Transaction

Recommended signature:

```sql
create_booking(
  p_room_id uuid,
  p_check_in date,
  p_check_out date,
  p_guest_count integer,
  p_idempotency_key uuid
)
```

The RPC returns at least:

```text
booking_id
booking_code
booking_status
payment_status
expires_at
unit_price
night_count
total_price
```

The entire function runs in one PostgreSQL transaction:

1. Require `auth.uid()`.
2. Validate `p_idempotency_key`.
3. Return the existing booking when the same user and idempotency key are submitted again.
4. Validate `check_in >= current_date`.
5. Validate `check_out > check_in`.
6. Validate maximum stay duration if the business defines one.
7. Lock the target `rooms` row with `FOR UPDATE` to serialize booking attempts for the same room type.
8. Validate the room exists, is active, and supports the guest count.
9. Opportunistically expire overdue `pending_payment` bookings for this room so delayed cron execution does not produce false “sold out” results.
10. Lock all `room_availability` rows for `[check_in, check_out)` in ascending date order.
11. Verify the number of rows equals the number of nights.
12. Verify every row has `available_units >= 1`.
13. Read the room price from the database and calculate totals server-side.
14. Insert the booking with a server-generated code and 30-minute expiry.
15. Decrement `available_units` by one for every consumed date.
16. Return the booking result.

If any validation or inventory update fails, raise an exception. PostgreSQL rolls back both the booking insert and every availability change.

The frontend must never provide `user_id`, booking status, payment status, unit price, total price, booking code, or expiry timestamp.

### 6.1 Why this prevents double booking

The room row lock serializes concurrent booking creation for the same room type. The second transaction waits, then reads availability after the first transaction commits. It cannot reserve inventory that has already reached zero.

The final inventory update must still include `available_units > 0`, and the RPC must assert that the updated row count equals `night_count`. This is a defensive second guard.

## 7. Expiration and Inventory Restoration

### 7.1 Single-booking function

Implement an internal or service-only function:

```sql
expire_booking(p_booking_id uuid)
```

Transaction behavior:

1. Lock the booking row using `FOR UPDATE`.
2. Return without changing inventory if it is already expired, cancelled, confirmed, or released.
3. Require:
   - `booking_status = 'pending_payment'`;
   - `payment_status IN ('unpaid', 'rejected')`;
   - `expires_at <= now()`;
   - `inventory_released_at IS NULL`.
4. Lock the related room row.
5. Increment availability for every date in `[check_in, check_out)`.
6. Use `least(total_units, available_units + 1)` as a defensive cap.
7. Set:
   - `booking_status = 'expired'`;
   - `expired_at = now()`;
   - `inventory_released_at = now()`.

`inventory_released_at` makes the operation idempotent. Calling expiration twice must not add inventory twice.

### 7.2 Batch expiration function

Implement a service-only batch function:

```sql
expire_pending_bookings(p_limit integer default 100)
```

Select due bookings using:

```text
booking_status = pending_payment
payment_status in unpaid/rejected
expires_at <= now()
inventory_released_at is null
```

Use `FOR UPDATE SKIP LOCKED` so multiple workers can run safely.

### 7.3 Scheduler

Preferred: Supabase Cron / PostgreSQL `pg_cron`, every minute.

```text
* * * * *  select public.expire_pending_bookings(100);
```

Fallback: a scheduled Supabase Edge Function invokes the service-only RPC with the service-role key.

The RPC must remain idempotent because scheduled jobs may retry.

### 7.4 Frontend countdown

The pending-payment page calculates:

```text
max(0, expires_at - current server-adjusted time)
```

When it reaches zero, the page refetches the booking. It does not update the status itself. The UI displays `expired` only after the database returns that state.

## 8. Payment and Expiration Race Handling

Payment approval/webhook and expiration can happen at the same moment. Both operations must lock the same booking row with `FOR UPDATE`.

- If verified payment obtains the lock first, it changes the booking to `confirmed`; expiration then does nothing.
- If expiration obtains the lock first, it changes the booking to `expired` and releases inventory; a later payment must not reconfirm it.
- A payment received after expiry is recorded as `late_payment` and sent to refund/manual-reconciliation handling.

Never restore inventory and confirm the same booking.

## 9. RPC Contracts

### 9.1 `create_booking()`

- Caller: authenticated guest.
- Uses `auth.uid()`.
- Creates booking and inventory hold atomically.

### 9.2 `upload_payment_proof()`

- Caller: booking owner.
- Requires booking not expired and `now() < expires_at`.
- Validates that the proof path belongs to the user and booking.
- Stores proof URL.
- Sets payment to `submitted` and booking to `payment_review`.
- Does not allow arbitrary status input.

### 9.3 `approve_payment()`

- Caller: admin or trusted payment webhook path.
- Locks booking row.
- Requires a valid transition from `payment_review` or `pending_payment`.
- Sets `payment_status = paid`, `booking_status = confirmed`, and `paid_at`.
- Does nothing on a repeated provider event or repeated approval.

### 9.4 `reject_payment()`

- Caller: admin.
- Locks booking row.
- Sets `payment_status = rejected`.
- Either:
  - returns to `pending_payment` with a new `expires_at = now() + 30 minutes`; or
  - cancels and releases inventory immediately.

The selected policy must be consistent in RPC, UI copy, and admin workflow.

### 9.5 `cancel_booking()`

- Caller: booking owner while cancellation is allowed, or admin.
- Locks booking row.
- Releases inventory exactly once.
- Sets booking to `cancelled`.
- Confirmed-booking cancellation rules must be handled separately from an unpaid cancellation.

### 9.6 `get_booking_by_code()` or booking read query

- Guest may only read their own booking.
- Admin may read any booking.
- Response includes authoritative status and `expires_at`.
- Expired codes return an expired booking, not “not found,” unless information-disclosure rules require a generic public response.

## 10. Security Implementation

### 10.1 Function security

For privileged RPC functions:

- Use `SECURITY DEFINER` only where required.
- Set a fixed search path, for example `SET search_path = public, pg_temp`.
- Schema-qualify tables and functions.
- Own functions with a non-login migration/admin role.
- Revoke execute from `PUBLIC` and `anon`.
- Grant guest RPCs only to `authenticated`.
- Grant expiry and webhook RPCs only to `service_role` or the scheduler owner.
- Check admin role inside admin RPCs; do not trust a frontend route guard.

### 10.2 RLS

Guests:

- Can select their own bookings and payments.
- Cannot directly insert or update `bookings`.
- Cannot directly update `room_availability`.
- Cannot approve, reject, confirm, expire, or release inventory.

Admins:

- Can read all bookings and payments.
- Perform state changes through approved RPCs.

Managers:

- Read-only access for reports.

`room_availability` may be publicly readable only if the UI needs availability search. Exact inventory counts may instead be exposed through a safe read RPC.

### 10.3 Payment proof storage

- Use a private bucket.
- Store files under a user/booking-scoped path.
- Validate ownership in storage policy and RPC.
- Use signed URLs for admin review.
- Restrict file type and size.

## 11. Stable Error Contract

RPC errors should map to stable application codes:

| Code | Meaning |
| --- | --- |
| `AUTH_REQUIRED` | No authenticated user |
| `ROOM_NOT_FOUND` | Room does not exist |
| `ROOM_INACTIVE` | Room cannot be booked |
| `INVALID_DATE_RANGE` | Dates are invalid |
| `GUEST_LIMIT_EXCEEDED` | Too many guests |
| `AVAILABILITY_NOT_INITIALIZED` | Date inventory rows are missing |
| `ROOM_UNAVAILABLE` | At least one night has no inventory |
| `BOOKING_EXPIRED` | Deadline has passed |
| `INVALID_STATUS_TRANSITION` | Operation is not allowed from current status |
| `PAYMENT_ALREADY_PROCESSED` | Duplicate or conflicting payment event |

The React app should map these codes to clear user-facing messages.

## 14. Required Test Matrix

| Scenario | Expected result |
| --- | --- |
| Two users reserve the last unit at the same time | Exactly one succeeds; the other receives `ROOM_UNAVAILABLE` |
| Same client retries the same idempotency key | Same booking returned; inventory decremented once |
| Same client uses a new key for already unavailable dates | Request rejected |
| Booking spans three nights | Three availability rows decremented |
| Check-out equals another booking's check-in | Both bookings allowed if all other nights are available |
| Booking expires | Status becomes expired and each consumed date is restored once |
| Expiry job runs twice | Inventory is not incremented twice |
| Guest cancels unpaid booking | Inventory restored once |
| Payment approval and expiry run concurrently | Booking ends either confirmed with held inventory or expired with released inventory, never both |
| Payment arrives after expiry | Booking remains expired; payment becomes late/manual reconciliation |
| Proof uploaded before deadline | Booking moves to payment review and is not expired as unpaid |
| Guest calls bookings REST insert/update | Denied by RLS/grants |
| Guest supplies another user's ID | Ignored or denied; `auth.uid()` is authoritative |
| Guest changes price in request | Ignored; total uses server-side price snapshot |
| Booking date availability row is missing | RPC fails and creates no booking |

## 15. Acceptance Criteria

The feature is complete only when:

- Double booking cannot be reproduced with concurrent database requests.
- Inventory cannot become negative or exceed total inventory.
- Booking creation and inventory decrement commit or roll back together.
- Unpaid bookings expire from database time after 30 minutes.
- Expiration and cancellation restore availability exactly once.
- Expired booking codes remain queryable as expired and cannot be paid/confirmed normally.
- Payment approval cannot race into an expired-and-released booking.
- React cannot directly insert/update booking status or room availability.
- Frontend countdown reflects `expires_at` and survives refresh/device changes.
- All RLS, RPC privilege, concurrency, and retry tests pass.

## 16. Recommended Execution Order

Execute in this order to avoid exposing a partially secured flow:

```text
Schema and constraints
  → transactional inventory RPCs
  → expiration/payment race handling
  → RLS and grants
  → scheduler
  → automated concurrency/security tests
  → frontend integration
  → controlled rollout and reconciliation
```

Do not switch the frontend to RPC booking creation until the schema, inventory release, RLS, scheduler, and concurrency tests are in place.

## 17. Verified Post-Stay Reviews

Rules:

- Only the authenticated owner of a `confirmed` / `paid` booking may review it.
- Rating opens when `current_date >= check_out` and is validated again by the database.

- Public review output never exposes `user_id`, email, phone, or booking ID.
